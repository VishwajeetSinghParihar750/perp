## Category A тАФ Test Failures (Confirmed at Runtime) ЁЯТЪ

### A1. `OrderBook` тАФ Market orders fail to match ЁЯТЪ

**File:** `apps/engine/src/classes/OrderBook.ts:147-245`
**Tests affected:** `should match a market buy against existing limit sell orders`, `should partially fill market buy if not enough liquidity`, `should match across multiple price levels on market buy`, `should match a market sell against existing limit buy orders`

**What happens:** A market order's `totalFilledQuantity` is always `0`. The inner while loop at line 177 does not guard against `orders.front()` returning `undefined` after all orders at a price level are consumed тАФ `frontOrder!.qty` throws `TypeError: Cannot read properties of undefined`. But even when corrected, the outer while loop at line 166 re-evaluates `oppositeSideOrders.empty()` but then calls `oppositeSideOrders.front()!` which may also be `undefined` because the `eraseElementByKey` at line 248 may not immediately report the map as empty.

**Root cause:** The inner while loop condition `currentOrder.filledQty < currentOrder.qty` (line 177) does not check `!orders.empty()`. After the last order at a price level is fully filled and popped, the loop continues, calling `orders.front()` on an empty `LinkList`, which returns `undefined`. The `!` non-null assertion hides the type error.

### A2. `LiquidationEngine` тАФ `getMarginRequired` does not throw on missing index price ЁЯТЪ

**File:** `apps/engine/src/classes/LiquidationEngine.ts:168`
**Test affected:** `should throw when no index price and no price given`

**What happens:** When no `price` is provided (MARKET order) and no `indexPrices[symbol]` has been set, `this.indexPrices[order.symbol]` is `undefined`. The `!` non-null assertion silences TypeScript, but at runtime `undefined * order.qty / 10` returns `NaN`. The test expects a thrown error but gets `NaN` instead.

### A3. `PositionManager` тАФ PnL calculation has wrong sign for position reductions ЁЯТЪ

**File:** `apps/engine/src/classes/PositionManager.ts:189-191`
**Test affected:** `should close position when qty reaches 0 and return PnL`

**What happens:** When reducing an existing LONG position (selling into it), `orderType` is `"SHORT"`, so the multiplier is `-1`:

```ts
(weighedAvgPrice - newPosition.price) *
  qtyForPnl *
  (orderType == "LONG" ? 1 : -1);
```

A profitable close (bought at 20, sold at 25) produces negative PnL because `(25-20) * 5 * (-1) = -25`. The multiplier should be based on **position type**, not order type. Additionally on full close, the entire margin (`100`) is added to PnL (line 213), inflating the value to `75` instead of the correct `25` (profit only).

## Category B тАФ Critical Runtime Bugs (Will Crash or Corrupt State)

### B1. `LiquidationEngine.loadSnapshot` тАФ No-op assignment discards all data ЁЯТЪ

**File:** `apps/engine/src/classes/LiquidationEngine.ts:64`

### B2. `LiquidationEngine.getLiquidationForPosition` тАФ LONG liquidation price is inverted ЁЯТЪ

**File:** `apps/engine/src/classes/LiquidationEngine.ts:95-97`

```ts
let newPrice =
  positon.price +
  (canTakeLoss / positon.qty) * (positon.type == "LONG" ? 1 : -1);
```

For LONG positions, this computes a liquidation price **above** the entry price. Example: entry=100, margin=1000, qty=10 => canTakeLoss=950 => liquidationPrice = `100 + 950/10 = 195`. **LONG positions should liquidate when price falls, not rises.** The formula should subtract for LONG: `entryPrice - (0.95 * margin) / qty`.

For SHORT the formula is correct (price rises to liquidate shorts).

**Consequence:** LONG positions are never liquidated on price drops. They would only be (incorrectly) liquidated when the price rallies (when they are profitable).

---

### B3. `LiquidationEngine.handleIndexPriceUpdate` тАФ Liquidated positions never removed from data structure ЁЯТЪ

**File:** `apps/engine/src/classes/LiquidationEngine.ts:142-151`

The while loop iterates `positionsMap.front()` but **never removes elements** (no `popFront`, no `eraseElementByKey`). This creates an infinite loop on the first price movement that triggers liquidation. Even if the loop somehow exited, the same positions would be liquidated again on every subsequent price update.

Additionally, `liquidation.started` / `liquidation.completed` events are **never emitted** by this function, so `Balances.lockAccount` is never called, and accounts are never locked during liquidation.

### B4. `EngineServer.handleClientRequsts` тАФ Unbounded recursion causes stack overflow ЁЯТЪ

**File:** `apps/engine/src/classes/EngineServer.ts:109`

```ts

```

Each processed message adds a stack frame via recursive call. JavaScript call stack limit is ~10k frames. Under sustained load this **will crash the engine**. Should use a `while(true)` loop.

### B7. `SnapshotManger` тАФ Lexicographic string comparison of Redis message IDs

**File:** `apps/engine/src/classes/SnapshotManger.ts:38,55-61`

Redis message IDs like `"1680000000000-0"` (13 digits) and `"999999999999-0"` (12 digits) are compared lexicographically with `<=` and `<`.
